Architecture Proposal Summary
1. Separate binary vs subcommand?
cmd/kanban-web/ (separate binary) vs kanban serve (subcommand). Recommendation: separate binary — clean separation, independent deployment.

Separate binary is the way to go

2. Web server permitted to commit+push?
Non-negotiable #4 says the CLI never auto-commits. Proposal: treat kanban-web as a separate execution context (like kanban ci-done) that IS allowed to commit+push. The CLI binary remains untouched.

Yes, the web server is allowed to commit + push.

3. Token storage — client-side cookie
HttpOnly, Secure, SameSite=Strict cookie with encrypted token. No server-side session store needed (zero budget). Server validates token against GitHub API on each write request. Stateless server.

Yes this sounds like a good solution

4. HTML templating — Go templates + vanilla JS
Go html/template (stdlib, no npm, no build step) + minimal vanilla JS for card detail modal. Templates embedded in binary via embed. Full page reload on form submission for MVP.

Im considering if htmx would be a good choice, then we do not
even have to use explicit javascript

5. Git sync — 60-second periodic pull
Goroutine runs git pull every 60 seconds. Pull-before-push on writes. "Last synced" indicator on the page. No webhook infrastructure needed.

I think this could be an environment variable or an argument
given to the program at start, which allows for experimenting
with the best frequency

6. Deployment — Caddy + systemd
Caddy for reverse proxy + auto Let's Encrypt HTTPS. systemd for process supervision. Deploy via scp binary + systemctl restart. Do you have a domain name? (Caddy needs one for auto-TLS, otherwise IP-only with self-signed certs.)
